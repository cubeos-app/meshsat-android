# 6. Android Keystore is the only home for client private keys

Date: 2026-05-18

## Status

Accepted

## Context

The pair-shell flow (spec/003) generates X25519 + Ed25519 client keypairs for mTLS with paired bridges. These private keys MUST be protected against:

- App package backup that ends up in operator's Google account.
- Operator-export of app data via Android's "Share via" flow.
- Debug pulls via `adb backup`.
- Accidental log statements during dev.

## Decision

All client private keys generated by the app go into the Android Keystore via `KeyPairGenerator.getInstance("EC", "AndroidKeyStore")`. The keystore is hardware-backed on most modern devices (StrongBox / TEE), and even on older devices the raw key bytes never leave the secure storage area.

Wrapper: `crypto/SecureKeyStore.kt`.

Raw key bytes shall NEVER appear in:
- `SharedPreferences` or `EncryptedSharedPreferences` (use Keystore directly)
- Room columns
- JSON exports
- Log statements (even at TRACE level)
- Backup payloads (set `android:allowBackup="false"` for the app OR exclude the Keystore alias from backup rules)

JWTs go in `EncryptedSharedPreferences` (AndroidX Security) — they're bearer tokens with bounded lifetime + can be re-issued, so the threat model is different.

## Consequences

**Positive:**
- StrongBox/TEE-backed key never leaves the secure enclave.
- `adb backup` cannot extract.
- Log scrubber regex can reasonably assert "no key bytes in logs."

**Negative:**
- Keystore aliases are device-bound. Operator reinstalling the app or factory-resetting loses pairings — must re-pair from the Bridge. Acceptable trade-off.

**Enforced by:** Component Article C-V (private keys) + Article C-VI (JWTs).
